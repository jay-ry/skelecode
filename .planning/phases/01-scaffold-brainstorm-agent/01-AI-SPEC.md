## 1. System Classification

**System Type:** Stateful Multi-Agent Workflow

**Description:**
The brainstorm agent is a stateful LangGraph pipeline that conducts a structured interview with users (solo devs, indie hackers, technical PMs, non-technical founders) to extract their project idea, then generates a comprehensive `project.md` specification. The pipeline has three nodes — extractor, reviewer, drafter — connected by conditional edges. The reviewer gates progress: if required fields (problem, users, features, stack, constraints) are missing, it returns control to the frontend to collect more input; once all fields are present, the drafter generates the full spec. "Good" means the generated `project.md` is complete, structured, and immediately actionable as input to the downstream sprint planner — a developer can hand it to any LLM and get a useful sprint plan without editing.

**Critical Failure Modes:**
1. **Infinite reviewer loop** — reviewer repeatedly flags missing_fields without ever reaching the drafter, hanging the user session
2. **Incomplete project.md** — drafter generates a spec missing required sections (Vision, Problem, Target Users, Core Features, Tech Stack, Data Model, Constraints), making downstream sprint planning fail
3. **SSE stream corruption or silence** — node events fail to yield or [DONE] sentinel is never emitted, leaving the frontend stuck with no feedback
4. **Reviewer misclassifies complete state** — reviewer says fields are missing when they are present, blocking the user from ever getting a spec
5. **Bot asks for info already provided** — extractor fails to carry forward conversation context, frustrating users who already answered a question

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / AI-assisted software planning

**User Population:** Three distinct personas with different levels of technical sophistication and different failure tolerances:
- **Solo devs / indie hackers** — technically fluent, time-constrained, typically starting greenfield projects alone. They use SkeleCode to skip the blank-page problem and get a structured plan before touching code. They will notice immediately if the spec contradicts their stated stack or invents features they never mentioned.
- **Technical PMs** — know enough to read a spec critically, but are not implementing it. They need user stories with testable acceptance criteria and sprint-scoped work. A vague spec with no definition of done is actively harmful to their planning process.
- **Non-technical founders** — have product intuition but cannot evaluate technical feasibility. They are most vulnerable to hallucinated tech choices or architectural assumptions that look reasonable but create downstream problems a developer will reject. They are least likely to catch errors before acting on the spec.

All three users interact conversationally via chat, then receive a `project.md` they may hand directly to a developer, another LLM, or SkeleCode's downstream sprint planner — often without deep review.

**Stakes Level:** Medium — a bad spec wastes developer time and cascades into poorly scoped sprints, but the user can review the output before acting on it. Non-technical founders have reduced ability to catch errors, raising their effective stakes level to High.

**Output Consequence:** The generated `project.md` becomes direct input to the sprint planner agent (Stage 2). A vague or incomplete spec produces sprints with undefined scope boundaries, untestable definitions of done, and missing constraints — wasting the first development session and requiring rework before any code is written. Errors in the spec compound: a hallucinated tech stack dependency in `project.md` can propagate through all sprint files and the skeleton generator.

### What Domain Experts Evaluate Against

A senior developer or experienced PM evaluating a requirements document uses the following criteria. Each dimension below is stated in practitioner language.

---

**Dimension: Section completeness**
- Good: All seven required sections are present and non-empty — Vision, Problem, Target Users, Core Features, Tech Stack, Data Model, Constraints. A developer can open the spec and immediately orient to the project without asking follow-up questions.
- Bad: One or more sections are missing, placeholder-filled ("TBD"), or collapsed into another section. The downstream sprint planner cannot generate a sprint if it cannot find a Tech Stack or Core Features section.
- Stakes: Critical
- Source: SkeleCode project spec (fixed template requirement); industry standard PRD completeness criteria (Atlassian, ProcessImpact quality requirements guidelines)

---

**Dimension: Feature specificity and scope boundary**
- Good: Each core feature is named, described in one sentence, and clearly inside the MVP boundary. Features that are explicitly out of scope are listed separately. A developer can look at the feature list and know what they are building in Sprint 1 without negotiating scope.
- Bad: Features described at product-category level ("user management", "notifications") with no specificity about what is and is not included. No out-of-scope list. The spec implies unlimited scope.
- Stakes: High
- Source: Practitioner knowledge — scope creep is consistently cited as the top cause of missed deadlines in solo/indie dev projects; GAIRE 2024 research on GenAI RE limitations identifies scope ambiguity as a primary failure mode

---

**Dimension: Tech stack feasibility and internal consistency**
- Good: The stated stack is internally consistent (e.g., Next.js frontend + FastAPI backend, not a mix of incompatible runtimes). Each stack choice maps to at least one stated core feature. If the user specified preferences, those preferences are honored exactly.
- Bad: Stack contains technologies the user never mentioned and did not ask for. Stack components conflict (e.g., two ORM libraries for the same database). A stated stack dependency does not exist or is misnamed (hallucinated package name).
- Stakes: High
- Source: LLM RE failure modes research (arxiv 2511.01324) — LLMs hallucinate package names and stack configurations when not grounded in user input; senior developer evaluation standard

---

**Dimension: User story presence and testability**
- Good: Each core feature has at least one user story in standard format ("As a [persona], I want [action] so that [outcome]") with a stated acceptance criterion that a developer can test in a browser without ambiguity.
- Bad: Features described from the system's perspective ("The system will support...") with no user story framing. Acceptance criteria are subjective ("works well", "performs acceptably") or absent entirely.
- Stakes: High
- Source: Agile/Scrum practitioner standard; Atlassian PRD guidelines; SkeleCode project spec requirement ("sprint-ready tickets with testable definition of done")

---

**Dimension: Constraint and assumption capture**
- Good: Technical constraints are explicit and attributed (e.g., "No OpenAI dependency — per project decision", "MVP desktop-only — mobile deferred"). Assumptions the spec is making are stated. A developer inheriting this spec knows what they cannot change.
- Bad: Constraints are absent or vague ("keep it simple"). Critical assumptions (auth provider, hosting platform, budget limits) are embedded as implied defaults rather than stated decisions. A developer must reverse-engineer constraints from the feature list.
- Stakes: Medium
- Source: ProcessImpact quality requirements — constraints must be explicitly captured to prevent requirement drift; SkeleCode context decisions document demonstrates the value of explicit constraint capture

---

**Dimension: Idea fidelity — spec matches what the user actually said**
- Good: The spec reflects the user's stated problem, users, and features without adding unrequested capabilities. If the user said "simple CRUD app", the spec does not introduce event sourcing or CQRS patterns. Technical terminology introduced by the AI matches the user's stated intent.
- Bad: The spec inflates the user's idea with capabilities they never mentioned. The problem statement has been generalized beyond what the user described. The target user population has been broadened to increase apparent scope.
- Stakes: High
- Source: Human-AI collaboration RE research (arxiv 2511.01324 — 81.2% of practitioners require humans to review and approve AI-generated requirements precisely because of idea drift); SkeleCode non-technical founder persona is particularly vulnerable to this failure

### Known Failure Modes in This Domain

These are domain-specific manifestations observed in AI-assisted requirements generation — not generic LLM hallucination.

1. **Scope inflation** — The agent interviews the user about a focused MVP and produces a spec that includes "nice to have" features the user mentioned in passing but never committed to. The spec reads like a Series A product, not an MVP. A developer working from it builds too much.

2. **Tech stack hallucination** — The drafter generates a stack that looks plausible but includes package names that do not exist, version combinations that are incompatible, or frameworks the user never specified. Non-technical founders cannot detect this; it reaches a developer who rejects the spec.

3. **Constraint amnesia** — The user stated a clear constraint during the conversation ("no auth in v1", "budget is zero so no paid APIs"). The extractor captured it, but the drafter's prompt does not carry it forward into the Constraints section. The generated spec contradicts the user's decision.

4. **User story collapse** — The drafter lists features but formats them as system capabilities rather than user stories. Acceptance criteria are absent or restate the feature name. A technical PM receives a spec that cannot be turned into sprint tickets without a full rewrite of every feature description.

5. **Context loss across conversation turns** — When a user provides information across multiple chat turns, the extractor aggregates correctly for the first few turns but drops earlier context as the conversation grows. The reviewer flags "missing_fields" for information the user already provided in turn 2 of 6. The user is asked to repeat themselves.

6. **Data model misalignment with features** — The drafter generates a Data Model section whose entities and relationships do not map to the stated Core Features. A feature requires a relationship that does not exist in the schema, or the schema includes tables for features that are out of scope. A developer cannot build the stated features from the stated schema without redesigning the model.

## 2. Framework Decision

**Selected Framework:** LangGraph

**Version:** 0.2.x (latest stable)

**Rationale:**
The brainstorm pipeline is a directed graph with conditional edges — extractor → reviewer → (drafter | END) — which is exactly what LangGraph was designed for. Its `astream()` API yields one event dict per node completion, mapping 1:1 to the required SSE event contract (`data: {"node": ..., "data": ...}\n\n`). `BrainstormState` as a `TypedDict` integrates cleanly with FastAPI's type system. The reviewer's `missing_fields` gate and 2-iteration loop cap are native graph edge constraints, not workarounds. No other framework handles conditional branching, per-node streaming, and Python-first stateful orchestration with the same depth.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| CrewAI | Flows architecture cannot express the reviewer's conditional branching as cleanly; checkpointing gaps surface when debugging the loop guard in production; 5.76x speed advantage is irrelevant when bottleneck is LLM latency |
| LangChain (without LangGraph) | Linear chains cannot express the conditional reviewer edge; no native per-node streaming to SSE |
| OpenAI Agents SDK | Hard constraint: no OpenAI dependency per project spec |
| Google ADK | Hard constraint: Gemini vendor lock-in unacceptable per project spec |

**Vendor Lock-In Accepted:** Partial — LangGraph is model-agnostic; Claude (claude-sonnet-4-5) is provider-locked per explicit project constraint (acceptable trade-off, documented in PROJECT.md)

### Entry Point Pattern
```python
# Minimal working BrainstormState + StateGraph — compiles and can be astream()ed
from typing import TypedDict, List, Literal
from langgraph.graph import StateGraph, START, END
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import SystemMessage, HumanMessage
import json


class BrainstormState(TypedDict):
    conversation: List[dict]   # [{role, content}, ...]
    extracted: dict            # {problem, users, features, stack, constraints}
    missing_fields: List[str]
    project_md: str
    status: str                # extracting | drafting | done


llm_extractor = ChatAnthropic(model="claude-sonnet-4-5", temperature=0.0, max_tokens=1024)
llm_drafter   = ChatAnthropic(model="claude-sonnet-4-5", temperature=0.7, max_tokens=4096)


async def extractor_node(state: BrainstormState) -> dict:
    # Build messages from conversation history
    messages = [SystemMessage(content="Extract project fields from the conversation.")]
    for msg in state["conversation"]:
        if msg["role"] == "user":
            messages.append(HumanMessage(content=msg["content"]))
        else:
            messages.append(SystemMessage(content=msg["content"]))
    response = await llm_extractor.ainvoke(messages)
    # In production: use with_structured_output(ExtractedFields) — see Section 4b
    extracted = {"problem": "...", "users": "...", "features": "...",
                 "stack": "...", "constraints": "..."}
    missing = [k for k, v in extracted.items() if not v or v == "..."]
    return {"extracted": extracted, "missing_fields": missing, "status": "extracting"}


async def reviewer_node(state: BrainstormState) -> dict:
    # Pure logic gate — no LLM call needed; routing happens in reviewer_router below
    return {}


async def drafter_node(state: BrainstormState) -> dict:
    messages = [
        SystemMessage(content="Write a project.md with sections: Vision, Problem, "
                               "Target Users, Core Features, Tech Stack, Data Model, Constraints."),
        HumanMessage(content=f"Project data:\n{json.dumps(state['extracted'], indent=2)}"),
    ]
    response = await llm_drafter.ainvoke(messages)
    return {"project_md": response.content, "status": "done"}


def reviewer_router(state: BrainstormState) -> Literal["drafter", "__end__"]:
    """Route to drafter if all fields present; route to END if any are missing."""
    return "drafter" if not state.get("missing_fields") else "__end__"


# Build and compile the graph — do this once at module load, reuse across requests
builder = StateGraph(BrainstormState)
builder.add_node("extractor", extractor_node)
builder.add_node("reviewer",  reviewer_node)
builder.add_node("drafter",   drafter_node)

builder.add_edge(START, "extractor")
builder.add_edge("extractor", "reviewer")
builder.add_conditional_edges(
    "reviewer",
    reviewer_router,
    {"drafter": "drafter", "__end__": END},  # map string keys to node names / END sentinel
)
builder.add_edge("drafter", END)

graph = builder.compile()


# FastAPI SSE endpoint — astream() yields {node_name: state_dict} per node completion
async def stream_brainstorm(conversation: List[dict]):
    init_state = BrainstormState(
        conversation=conversation,
        extracted={},
        missing_fields=[],
        project_md="",
        status="extracting",
    )
    # stream_mode="updates" yields only the keys each node changed — not the full state blob
    async for event in graph.astream(init_state, stream_mode="updates"):
        node_name = list(event.keys())[0]
        data      = list(event.values())[0]
        yield f"data: {json.dumps({'node': node_name, 'data': data})}\n\n"
    yield "data: [DONE]\n\n"
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `StateGraph(State)` | Directed graph whose nodes read and write a shared TypedDict; the root builder object | Every LangGraph workflow — call `.compile()` after all nodes and edges are added |
| `TypedDict` state | Shared data container passed between all nodes; nodes return partial dicts that are merged | Define once in `models/brainstorm_state.py`; nodes only return the keys they changed |
| `add_conditional_edges(src, fn, map)` | Attaches a routing function to a node; `fn(state)` returns a string key; `map` resolves it to the next node name or `END` | Wherever a node must branch — here the reviewer routes to `"drafter"` or `"__end__"` based on `missing_fields` |
| `astream(input, stream_mode="updates")` | Async generator yielding `{node_name: changed_state_keys}` after each node completes | Use in the FastAPI SSE endpoint; `"updates"` mode gives one small diff per node, ideal for per-node SSE events |
| `END` (sentinel constant) | Special value that terminates graph execution when an edge points to it | Reviewer routes to `END` when fields are missing (frontend collects more input); drafter routes to `END` after generating the spec |

### Common Pitfalls

1. **Returning full state from a node instead of partial updates.** LangGraph merges dicts — nodes should return only the keys they changed (e.g., `{"extracted": {...}, "missing_fields": [...]}` not the full `BrainstormState`). Returning the full state overwrites all keys unnecessarily and makes `stream_mode="updates"` events large and noisy.

2. **Using `stream_mode="values"` when you want per-node SSE events.** `stream_mode="values"` yields the entire accumulated state after every node — you get a full state blob, not a diff. Use `stream_mode="updates"` for SSE where the frontend needs to know which node just ran and what it changed.

3. **Defining a sync node function and calling an async LLM inside it.** If a node is `def` (not `async def`), calling `await model.ainvoke(...)` is a syntax error. All three nodes in this pipeline call `ChatAnthropic` — declare them `async def` and use `await model.ainvoke(...)` throughout.

4. **Calling `asyncio.run()` inside a FastAPI route.** FastAPI routes run inside uvicorn's existing event loop. Wrapping `graph.astream()` in `asyncio.run()` raises `RuntimeError: This event loop is already running`. Declare the route handler `async def` and use `async for` directly — no `asyncio.run()` needed.

5. **Forgetting to map routing function return values to node names in `add_conditional_edges`.** The routing function must return string keys. `END` is a sentinel object, not a string — you must pass the explicit mapping dict: `{"drafter": "drafter", "__end__": END}`. Returning the `END` object directly from the routing function without a map raises a `KeyError` at compile time in LangGraph 0.2.x.

6. **No loop guard when the reviewer can route back to extractor.** This design routes reviewer to `END` (not back to extractor) when fields are missing, so infinite looping is impossible within one graph run. However, if you later change the design to loop internally, add a `iteration_count: int` field to `BrainstormState` and cap it at 2 in the reviewer router — LangGraph has no built-in loop limit.

## 4. Implementation Guidance

**Model Configuration:**

Use two separate `ChatAnthropic` instances — one cold (extractor), one warm (drafter). The reviewer node requires no LLM call.

```python
from langchain_anthropic import ChatAnthropic

# Extractor: deterministic structured extraction — temperature 0.0
llm_extractor = ChatAnthropic(
    model="claude-sonnet-4-5",
    temperature=0.0,
    max_tokens=1024,   # structured JSON output — never needs more than ~500 tokens
)

# Drafter: generative prose — temperature 0.7 for varied but coherent output
llm_drafter = ChatAnthropic(
    model="claude-sonnet-4-5",
    temperature=0.7,
    max_tokens=4096,   # full project.md with 7 sections; budget headroom for verbose specs
)
```

`claude-sonnet-4-5` is the model ID for this project per project spec. Do not substitute `claude-3-5-sonnet-20241022` — use the `claude-sonnet-4-5` identifier. Verify current model IDs at https://docs.anthropic.com/en/docs/about-claude/models if the alias fails.

**Core Pattern:**

Three-node graph with conditional reviewer routing. The reviewer is a pure logic gate — no LLM call, zero API cost, zero latency.

```python
# agents/brainstorm.py
from typing import TypedDict, List, Literal
from pathlib import Path
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import SystemMessage, HumanMessage
from langgraph.graph import StateGraph, START, END
from models.brainstorm_state import BrainstormState, ExtractedFields
import json

# Load system prompts at module startup — not inside node functions
EXTRACTOR_PROMPT = Path("prompts/extractor_system.txt").read_text()
DRAFTER_PROMPT   = Path("prompts/drafter_system.txt").read_text()

llm_extractor = ChatAnthropic(model="claude-sonnet-4-5", temperature=0.0, max_tokens=1024)
llm_drafter   = ChatAnthropic(model="claude-sonnet-4-5", temperature=0.7, max_tokens=4096)

# with_structured_output forces Claude to return a valid ExtractedFields object via tool-calling
structured_extractor = llm_extractor.with_structured_output(ExtractedFields)

REQUIRED_FIELDS = ["problem", "users", "features", "stack", "constraints"]


async def extractor_node(state: BrainstormState) -> dict:
    """Read full conversation history, extract project fields, identify gaps."""
    messages = [SystemMessage(content=EXTRACTOR_PROMPT)]
    for msg in state["conversation"]:
        cls = HumanMessage if msg["role"] == "user" else SystemMessage
        messages.append(cls(content=msg["content"]))

    result: ExtractedFields = await structured_extractor.ainvoke(messages)
    extracted = result.model_dump(exclude_none=True)
    missing = [f for f in REQUIRED_FIELDS if not extracted.get(f)]
    return {
        "extracted": extracted,
        "missing_fields": missing,
        "status": "extracting",
    }


async def reviewer_node(state: BrainstormState) -> dict:
    """Pure logic gate — validates state, returns no changes. Routing is in reviewer_router."""
    return {}


async def drafter_node(state: BrainstormState) -> dict:
    """Generate project.md from the extracted fields dict."""
    messages = [
        SystemMessage(content=DRAFTER_PROMPT),
        HumanMessage(content=f"Project data:\n{json.dumps(state['extracted'], indent=2)}"),
    ]
    response = await llm_drafter.ainvoke(messages)
    return {"project_md": response.content, "status": "done"}


def reviewer_router(state: BrainstormState) -> Literal["drafter", "__end__"]:
    return "drafter" if not state.get("missing_fields") else "__end__"


# Compile once at module import — safe to reuse across concurrent FastAPI requests
# Each request creates a new BrainstormState; the compiled graph object is stateless
builder = StateGraph(BrainstormState)
builder.add_node("extractor", extractor_node)
builder.add_node("reviewer",  reviewer_node)
builder.add_node("drafter",   drafter_node)
builder.add_edge(START, "extractor")
builder.add_edge("extractor", "reviewer")
builder.add_conditional_edges(
    "reviewer",
    reviewer_router,
    {"drafter": "drafter", "__end__": END},
)
builder.add_edge("drafter", END)

graph = builder.compile()
```

**Tool Use:**

None required for Phase 1. The pipeline is a pure LLM workflow — no web search, file I/O, or tool calls. Do not add LangChain tool infrastructure until Phase 2+ requires it. Unused tool scaffolding increases token overhead and debugging surface area without benefit.

**State Management:**

`BrainstormState` is stateless between HTTP requests. The frontend holds the full conversation in React `useState` and re-sends it on every POST to `/api/brainstorm`. There is no server-side session, no LangGraph checkpointer, and no database this phase. Each request creates a fresh `BrainstormState` from the incoming conversation list. This means:
- No `thread_id` configuration needed on `astream()` calls
- No `MemorySaver` or `AsyncRedisSaver` checkpointer
- No `config={"configurable": {"thread_id": "..."}}` argument

If session persistence is added in a future phase, insert a `MemorySaver` checkpointer at compile time and pass thread IDs at call time — it is a one-line change to the `compile()` call.

**Context Window Strategy:**

The extractor reads the full `conversation` list on every call. Claude Sonnet 4.5 has a 200k token context window — essentially unbounded for a brainstorm session of typical length (5-15 exchanges). Set a defensive soft cap at the API layer: if `len(conversation) > 50`, truncate to the last 50 messages before building `BrainstormState`. This prevents runaway sessions from hitting API limits in edge cases.

The drafter receives only `extracted{}` (~5 fields, approximately 200 tokens) — it never sees the raw conversation. This is the correct design: the extractor acts as a compression layer, and the drafter's cost stays constant regardless of conversation length.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

Use `ChatAnthropic.with_structured_output()` for the extractor node. This forces Claude to return a valid Pydantic object via its tool-calling mechanism, eliminating JSON parsing errors and providing automatic field validation.

**Define the output schema in `models/brainstorm_state.py`:**

```python
# models/brainstorm_state.py
from typing import TypedDict, List, Optional
from pydantic import BaseModel, Field


class ExtractedFields(BaseModel):
    """Structured output from the extractor node.
    All fields are Optional — the LLM may not have enough conversation context yet.
    missing_fields is computed from None values, not returned by the LLM.
    """
    problem: Optional[str] = Field(
        None,
        description="The core problem the project solves, in one or two sentences."
    )
    users: Optional[str] = Field(
        None,
        description="The primary target users — who they are and their key distinguishing trait."
    )
    features: Optional[str] = Field(
        None,
        description="The 3-5 core MVP features as a comma-separated list."
    )
    stack: Optional[str] = Field(
        None,
        description="Preferred tech stack: frontend, backend, database. Use 'undecided' if not mentioned."
    )
    constraints: Optional[str] = Field(
        None,
        description="Budget, timeline, team size, or technical constraints. Use 'none stated' if absent."
    )


class BrainstormState(TypedDict):
    conversation: List[dict]   # [{role: str, content: str}, ...]
    extracted: dict            # populated from ExtractedFields.model_dump(exclude_none=True)
    missing_fields: List[str]  # fields that are still None in ExtractedFields
    project_md: str
    status: str                # extracting | drafting | done
```

**Wire into the extractor node with retry logic:**

```python
from langchain_anthropic import ChatAnthropic
from pydantic import ValidationError
import logging

logger = logging.getLogger(__name__)

llm_extractor = ChatAnthropic(model="claude-sonnet-4-5", temperature=0.0, max_tokens=1024)
structured_extractor = llm_extractor.with_structured_output(ExtractedFields)

REQUIRED_FIELDS = ["problem", "users", "features", "stack", "constraints"]
MAX_RETRIES = 2


async def extractor_node(state: BrainstormState) -> dict:
    messages = build_extractor_messages(state["conversation"])  # see Section 4

    for attempt in range(MAX_RETRIES + 1):
        try:
            result: ExtractedFields = await structured_extractor.ainvoke(messages)
            extracted = result.model_dump(exclude_none=True)
            missing = [f for f in REQUIRED_FIELDS if not extracted.get(f)]
            return {
                "extracted": extracted,
                "missing_fields": missing,
                "status": "extracting",
            }
        except (ValidationError, Exception) as exc:
            logger.warning(
                "extractor_node attempt %d/%d failed: %s",
                attempt + 1, MAX_RETRIES + 1, exc,
            )
            if attempt == MAX_RETRIES:
                # Graceful degradation — treat all fields as missing, let frontend ask follow-up
                return {
                    "extracted": {},
                    "missing_fields": REQUIRED_FIELDS[:],
                    "status": "extracting",
                }
```

Note: `with_structured_output()` on Anthropic models uses Claude's tool-calling API under the hood. If Claude fails to invoke the tool correctly, LangChain raises a `ValidationError` or returns `None` — catch both. Log the attempt number and exception so you can distinguish transient failures from systematic prompt issues. Do not catch silently.

### Async-First Design

**How async works in LangGraph for this pipeline:**

LangGraph node functions can be sync (`def`) or async (`async def`). When `astream()` is called, LangGraph runs async nodes directly with `await` and wraps sync nodes in a thread pool. Because all three nodes in this pipeline call `ChatAnthropic.ainvoke()`, declare every node `async def` — mixing sync and async within a node is not possible.

```python
# CORRECT — async node with async LLM call
async def extractor_node(state: BrainstormState) -> dict:
    result = await structured_extractor.ainvoke(messages)
    return {"extracted": result.model_dump(exclude_none=True), "missing_fields": [...]}

# WRONG — sync node calling sync LLM — blocks the event loop during the API call
def extractor_node(state: BrainstormState) -> dict:
    result = structured_extractor.invoke(messages)  # blocks uvicorn's event loop
    return {"extracted": result.model_dump(exclude_none=True)}
```

**The one common mistake — `asyncio.run()` inside a FastAPI route:**

FastAPI runs inside uvicorn's event loop. Calling `asyncio.run(graph.astream(...))` inside any FastAPI route handler raises `RuntimeError: This event loop is already running`. The correct pattern is an `async def` route handler using `async for`:

```python
# CORRECT — async route handler, async for loop
@router.post("/api/brainstorm")
async def brainstorm_endpoint(req: BrainstormRequest):
    conversation = req.conversation[-50:]  # defensive truncation
    init_state = BrainstormState(
        conversation=conversation, extracted={},
        missing_fields=[], project_md="", status="extracting",
    )

    async def event_stream():
        async for event in graph.astream(init_state, stream_mode="updates"):
            node_name = list(event.keys())[0]
            data      = list(event.values())[0]
            yield f"data: {json.dumps({'node': node_name, 'data': data})}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")


# WRONG — asyncio.run() inside an async context raises RuntimeError
@router.post("/api/brainstorm")
async def brainstorm_endpoint(req: BrainstormRequest):
    result = asyncio.run(graph.astream(init_state))  # RuntimeError: loop already running
```

**Stream vs. await for this pipeline:**

- Use `astream()` at the graph level in the SSE endpoint — the frontend needs per-node progress events.
- Use `ainvoke()` inside individual nodes when calling the LLM — nodes return a complete result dict, not a token stream.
- Do not attempt token-level streaming through SSE in Phase 1. Token streaming from inside nodes requires `get_stream_writer()` and `stream_mode="custom"`, which adds significant complexity for marginal UX benefit. The drafter node generates the full `project_md` and the frontend receives it as a single SSE event.

### Prompt Engineering Discipline

Each node has a distinct concern — never reuse or share system prompts across nodes.

**Extractor system prompt** (`prompts/extractor_system.txt`):
```
You are a project requirements extractor. Your only job is to read a conversation
between a user and a brainstorm interviewer and extract structured project information.

Extract ONLY what has been explicitly stated — do not infer, fill in gaps, or make assumptions.
If a field has not been mentioned in the conversation, return null for that field.

Fields to extract:
- problem: the core problem the project solves
- users: the primary target users
- features: the 3-5 core MVP features
- stack: the preferred tech stack
- constraints: any stated budget, timeline, or technical constraints
```

**Drafter system prompt** (`prompts/drafter_system.txt`):
```
You are a technical project specification writer. Given extracted project data,
write a comprehensive project.md with these sections in exact order:

# Vision
# Problem
# Target Users
# Core Features
# Tech Stack
# Data Model
# Constraints

Rules:
- Every section must be populated — never leave a section empty or write "TBD"
- Core Features: write each feature as a user story with an acceptance criterion
- Tech Stack: list only technologies from the provided data; do not invent dependencies
- Data Model: define entities and relationships that support exactly the listed Core Features
- Constraints: include all constraints from the provided data verbatim
```

**Rules:**
- Load system prompts from files (`Path("prompts/extractor_system.txt").read_text()`) at module startup, not inside node functions. File loading inside a node runs on every invocation and is wasteful.
- Set `max_tokens` explicitly on every `ChatAnthropic` instance. Never leave it unbounded in production — a verbose drafter can consume 10x expected tokens on a long spec.
- The reviewer node requires no LLM call in this design. If you add an LLM-based reviewer in a future iteration, give it its own focused prompt — not a copy of the extractor prompt.
- Few-shot examples are not needed for extraction with `with_structured_output()` — the Pydantic field descriptions act as the implicit example. For the drafter, if output quality is inconsistent, add one complete `project.md` example inline in the drafter system prompt.

### Context Window Management

Claude Sonnet 4.5 has a 200k token context window. A typical brainstorm session is 5-15 exchanges (~1,000-4,000 tokens total). There is no risk of hitting the context limit in normal usage.

**Defensive truncation — enforced at the API layer, not inside the graph:**

```python
MAX_MESSAGES = 50  # ~10,000 tokens worst case — well within the 200k window

@router.post("/api/brainstorm")
async def brainstorm_endpoint(req: BrainstormRequest):
    # Truncate from the front — preserve the most recent context
    conversation = req.conversation[-MAX_MESSAGES:]
    ...
```

**If conversation grows beyond 50 messages (future-proofing for power users):**

Implement rolling summarization before truncation:
1. Summarize messages `[0..N-20]` into a single summary string using a cheap model call
2. Prepend as `{"role": "system", "content": "Conversation summary: " + summary}`
3. Append messages `[N-20..N]` verbatim

This preserves early context (the original project idea) while keeping token count bounded. Do not implement for Phase 1 — the 50-message hard truncation is sufficient.

**The drafter never sees the raw conversation.** It only receives `extracted{}` (~5 fields, approximately 200 tokens). This architectural decision keeps drafter cost constant regardless of conversation length and eliminates the risk of the drafter drifting toward conversation tone over spec structure.

## 5. Evaluation Strategy

### Dimensions

The first six dimensions map directly to the domain rubric ingredients established in Section 1b. The final three are technical dimensions specific to the LangGraph multi-agent architecture.

| Dimension | PASS | FAIL | Measurement Approach | Priority |
|-----------|------|------|---------------------|----------|
| **Section completeness** | All 7 sections present and non-empty: Vision, Problem, Target Users, Core Features, Tech Stack, Data Model, Constraints. No section contains "TBD" or is a copy of its heading. | Any section is absent, contains only "TBD", or is collapsed into another section (e.g., Constraints folded into Tech Stack). | Code — regex check for each `# Section` heading + non-empty body assertion. Run on every generated `project_md`. | Critical |
| **Feature specificity and scope boundary** | Each feature entry includes a name, a one-sentence description of what is included, and either an explicit out-of-scope list or a scoping phrase ("MVP only", "not in v1"). A developer can read the list and know exactly what to build in Sprint 1. | Features described at category level only ("user management") with no inclusion/exclusion boundary. No out-of-scope list present anywhere in the spec. | LLM Judge — prompt: "For each feature listed in Core Features, does it include a specific description of what is in scope? Is there any signal about what is out of scope? Answer per-feature, then give an overall pass/fail." | High |
| **Tech stack feasibility and internal consistency** | Stack section lists only technologies the user mentioned or explicitly agreed to. No package name appears that is not a real, publicly available library. All stack components are compatible with each other (e.g., no two ORMs for the same database). | Stack contains a technology the user never mentioned. A listed package name does not exist (hallucination). Two incompatible runtime choices appear together (e.g., Django ORM and SQLAlchemy for the same PostgreSQL instance). | LLM Judge — prompt: "Given the conversation transcript and the generated Tech Stack section, identify any technologies that were not mentioned by the user or are hallucinated package names. Identify any compatibility conflicts. Output: list of violations or 'none'." Human spot-check on 5% sample for hallucinated package names. | High |
| **User story presence and testability** | Every feature in Core Features has at least one user story in "As a [persona], I want [action] so that [outcome]" format. Each story has an acceptance criterion that names a concrete, browser-observable outcome (not "works correctly" or "is fast"). | Any feature described as "The system will..." with no user story framing. Acceptance criteria present but subjective ("performs acceptably", "loads quickly"). Acceptance criteria absent entirely. | LLM Judge — prompt: "For each feature in Core Features: (1) Is there a user story in As/I want/so that format? (2) Does the acceptance criterion describe a specific observable browser state a developer could verify? Score each feature 0 or 1 on both dimensions. Overall pass requires all features scoring 1 on both." | High |
| **Constraint and assumption capture** | Constraints section explicitly lists at least one constraint with attribution (e.g., "No OpenAI dependency — per project decision"). Assumptions are stated as assumptions, not embedded as implicit defaults. A developer reading the spec can identify what is fixed and what is open. | Constraints section is empty or contains only "none". Constraints mentioned during the conversation (captured in `extracted.constraints`) do not appear verbatim in the generated spec. Critical assumptions about auth, hosting, or budget are absent. | Code — assert that `extracted["constraints"]` content appears in the Constraints section of `project_md` (substring match). LLM Judge for assumption surfacing quality. | Medium |
| **Idea fidelity** | The spec's Problem statement matches the user's stated problem without generalization. Core Features contains only capabilities the user named or explicitly accepted. The target user population is not broadened beyond what the user described. | Problem statement elevated from the user's specific context to a broad market framing ("millions of users face..."). Features present that the user mentioned only in passing or as "maybe someday". User population broadened (user said "solo devs", spec says "developers and product teams"). | LLM Judge — prompt: "Compare the conversation transcript against the generated spec. For each of these dimensions: (1) Does the Problem match what the user actually described? (2) Does Core Features contain only what the user stated or agreed to? (3) Is the target user population what the user described? Flag any drift with a quoted example from both the conversation and the spec." Human review on all non-technical founder sessions (highest fidelity risk). | High |
| **State transition correctness** | On every request, the node sequence in the SSE stream is exactly: `extractor` event, then `reviewer` event, then (if fields complete) `drafter` event. No node fires out of order. Drafter never fires when `missing_fields` is non-empty. | `drafter` event appears in stream when `missing_fields` was non-empty at reviewer. `reviewer` event absent from stream. Nodes appear in wrong order (e.g., `drafter` before `reviewer`). | Code — parse SSE event log from test run, assert node sequence matches expected DAG. pytest fixture captures all `node` keys from the stream and asserts order. | Critical |
| **Reviewer loop guard** | When `BrainstormState.iteration_count` reaches 2, the reviewer router forces the route to `drafter` regardless of `missing_fields`. The graph never emits more than 2 `reviewer` events in a single session. | Any test session produces more than 2 `reviewer` events in the SSE stream. Graph hangs without emitting `[DONE]` when fields are perpetually missing. | Code — unit test with a synthetic `BrainstormState` where `missing_fields` is always non-empty; assert `[DONE]` is emitted within 2 reviewer passes and `drafter` fires on the second pass. | Critical |
| **SSE stream completeness** | Every graph invocation ends with `data: [DONE]\n\n` as the final SSE event. No stream terminates silently (connection closes without `[DONE]`). `[DONE]` is emitted even when the reviewer routes to `END` without calling the drafter. | Stream closes without a `[DONE]` event. `[DONE]` appears mid-stream before graph execution is complete. Connection hangs for more than 30 seconds after the last node event. | Code — integration test asserts the last line of every captured SSE stream is `data: [DONE]`. Timeout test asserts connection closes with error event (not silent) after 30s inactivity. | Critical |

- **Section completeness** — automated (Code): section presence check runs without human involvement.

- **Tech stack feasibility** — LLM judge with mandatory human spot-check: a senior developer reviews all 20 dataset examples for this dimension before the dataset is locked. Tech hallucination is too subtle for an uncalibrated LLM judge to catch reliably.
- **Idea fidelity** — human primary for all non-technical founder persona examples (highest fidelity risk); LLM judge for solo dev and technical PM examples with 20% human spot-check.

**Creation timeline:** Dataset construction begins during implementation week, not after. Write at least 5 examples (one per category) before writing the first eval test. The dataset informs the test suite; building them in the wrong order produces tests that pass by coincidence.

## 6. Guardrails

### Online (Real-Time)

These guardrails run on every request. Each adds minimal latency — pure Python checks against existing state, no additional LLM calls.

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Reviewer loop cap** | `BrainstormState.iteration_count >= 2` — reviewer has already returned to frontend twice without reaching the drafter | Force-route to drafter with current `extracted` dict, regardless of `missing_fields`. Drafter receives an explicit instruction in its prompt: "Some fields may be incomplete — make reasonable inferences from available data rather than leaving sections empty." Emit a `data: {"node": "reviewer", "data": {"forced": true, "reason": "iteration_cap"}}` SSE event so the frontend can optionally display a notice. |
| **project.md section validator** | After drafter completes and `project_md` is set: check that all 7 required section headings (`# Vision`, `# Problem`, `# Target Users`, `# Core Features`, `# Tech Stack`, `# Data Model`, `# Constraints`) are present using regex match | If any heading is missing: retry drafter once with an explicit additional instruction appended to the prompt: "The following sections were missing from your previous output — you MUST include them: [list of missing sections]." Log `drafter_retry=True` for monitoring. If second attempt still fails, emit the partial spec with a `data: {"node": "drafter", "data": {"partial": true, "missing_sections": [...]}}` SSE event. |
| **SSE stream timeout** | No SSE event emitted for > 30 seconds during an active graph run (detected via asyncio timeout wrapper around `astream()`) | Emit `data: {"node": "error", "data": {"reason": "stream_timeout", "message": "Generation timed out — please try again"}}\n\n` then `data: [DONE]\n\n`. Close the connection cleanly. Log `sse_timeout=True`. Never let the connection hang silently. |
| **Conversation length cap** | `len(req.conversation) > 50` messages on incoming request | Truncate to last 50 messages before building `BrainstormState` — preserve the most recent context. This is a pre-graph guardrail, not a graph node. Log `conversation_truncated=True` with original length for monitoring. |

### Offline (Flywheel)

These signals are computed asynchronously after sessions complete. They drive prompt iteration and model improvement without adding request latency.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|-----------------------|
| **section_completeness_rate** — % of generated `project_md` files containing all 7 required sections | Computed automatically on 100% of sessions (code check, zero cost) | If rate drops below 85% over any rolling 7-day window: alert (see Section 7). Investigate drafter prompt — likely cause is a formatting regression or a new conversation pattern the drafter prompt does not handle. |
| **reviewer_loop_rate** — % of sessions where the reviewer loop cap was triggered (iteration_count reached 2) | Computed automatically on 100% of sessions via `forced=True` log events | If rate exceeds 20% over any rolling 7-day window: alert. Indicates the extractor is systematically failing to extract fields even when users have provided them — likely a prompt regression or a new conversation pattern (e.g., technical jargon the extractor does not recognize as a stack preference). |
| **idea_fidelity_weekly_sample** — human-reviewed score of whether the generated spec matches what the user described | 5% random sample of all sessions weekly, prioritized toward non-technical founder sessions (identified by low technical term density in conversation messages) | If average human fidelity score drops below 3.5/5.0 over two consecutive weeks: review drafter prompt for scope inflation patterns. Consider adding a few-shot example that demonstrates restraint. |
| **drafter_retry_rate** — % of sessions where the section validator triggered a drafter retry | Computed automatically on 100% of sessions | If rate exceeds 10%: the drafter is consistently missing sections. Investigate whether a specific section (e.g., Data Model) is the recurring miss — add explicit reinforcement for that section in the drafter system prompt. |
| **sse_error_rate** — % of streams that terminate without emitting `[DONE]` (detected via frontend error logging or backend exception capture) | Computed on 100% of sessions via log aggregation | If rate exceeds 5%: immediate alert. SSE failures are user-visible — they see a blank preview pane with no feedback. Investigate connection handling and timeout behavior. |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix (self-hosted, open-source) — deployed as a Railway sidecar service alongside the FastAPI backend. Phoenix receives OpenTelemetry spans from LangGraph node executions via the OTLP exporter configured at FastAPI startup. UI available at the internal Railway service URL (not public-facing). No Phoenix cloud account required.

**Deployment note:** Add `arize-phoenix` as a separate Railway service using the official Docker image (`arizephoenix/phoenix:latest`). Set `PHOENIX_PORT=6006`. Point the FastAPI backend's `OTLP_ENDPOINT` env var to the internal Railway service URL.

### Key Metrics to Track

| Metric | Description | Collection Method |
|--------|-------------|-------------------|
| `section_completeness_rate` | Percentage of generated `project_md` outputs containing all 7 required section headings | Code check runs post-drafter in the section validator guardrail; result logged as structured JSON to Railway logs and forwarded to Phoenix as a span attribute |
| `reviewer_loop_rate` | Percentage of requests where the reviewer loop cap fired (`iteration_count` reached 2) | Logged as `reviewer_forced=True` span attribute on the reviewer node span in Phoenix |
| `p50_brainstorm_duration` / `p95_brainstorm_duration` | Wall-clock time from first SSE event to `[DONE]` event | Phoenix captures span start/end times for the full graph execution automatically via LangChain instrumentation |
| `sse_error_rate` | Percentage of streams that terminate without emitting `[DONE]` | Backend exception handler logs `sse_error=True` to Railway logs; frontend logs missing `[DONE]` events and POSTs to a `/api/telemetry` error endpoint |
| `drafter_retry_rate` | Percentage of drafts that fail section validation and require a second drafter call | Logged as `drafter_retry=True` span attribute on the drafter node span |

### Alert Thresholds

| Condition | Threshold | Alert Channel | Severity |
|-----------|-----------|---------------|----------|
| `section_completeness_rate` drops below 85% | Rolling 7-day average | Email (Railway alert or Resend) | High — spec quality is degraded; downstream sprint planner will receive incomplete input |
| `sse_error_rate` exceeds 5% | Rolling 24-hour window | Email — immediate | Critical — users see blank preview panes; core value loop is broken |
| `reviewer_loop_rate` exceeds 20% | Rolling 7-day average | Email | High — extractor regression; users are being asked to repeat information they already provided |
| `p95_brainstorm_duration` exceeds 60 seconds | Rolling 24-hour window | Email | High — Claude API timeout or network issue; user experience is degraded |
| `drafter_retry_rate` exceeds 10% | Rolling 7-day average | Email | Medium — drafter prompt is producing incomplete sections systematically |

### Smart Sampling Strategy

Three tiers of human review, applied after sessions complete:

**Tier 1 — Always review (signal-based, zero sampling rate — every qualifying session):**
- Sessions where the reviewer loop cap fired (`reviewer_forced=True`)
- Sessions where the drafter section validator triggered a retry (`drafter_retry=True`)
- Sessions that ended with an SSE error (`sse_error=True`)
- Sessions from users whose conversation message history contains fewer than 3 technical terms per 10 messages (proxy for non-technical founder persona — highest idea fidelity risk)

**Tier 2 — Random sample (5% of all sessions, weekly batch):**
- Select 5% of sessions uniformly at random each week
- Reviewer: senior developer for section completeness and tech stack feasibility; PM for user story testability
- Goal: catch systematic quality drift not captured by the code-based metrics

**Tier 3 — Targeted deep review (triggered by metric alerts):**
- When `idea_fidelity_weekly_sample` human scores average below 3.5/5.0 for two consecutive weeks: pull 20 sessions from the prior two weeks and review specifically for scope inflation and constraint amnesia
- When `reviewer_loop_rate` alert fires: pull the 10 most recent loop-cap sessions and inspect the conversation transcripts to identify what field the extractor is failing to capture

**Non-technical founder identification heuristic:**
Count technical terms per message using a fixed vocabulary list (framework names, database names, API terminology, cloud provider names). Sessions averaging fewer than 0.3 technical terms per user message are flagged for Tier 1 review. This heuristic is intentionally coarse — false positives (technical users writing concisely) are acceptable because the cost of over-reviewing is low and the cost of missing a fidelity failure for a non-technical founder is high.
