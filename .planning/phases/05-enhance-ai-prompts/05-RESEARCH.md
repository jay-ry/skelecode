## Summary

Phase 5 is a focused prompt engineering and rendering update — no schema changes, no new agents, no new dependencies. The core shift is replacing the structured-output sprint planner with a free-text markdown planner that produces richer documents matching the `plan/sprint-1.md` reference format. This change also requires a frontend update to render `content_md` markdown in `SprintCard`.

The existing codebase already has the correct foundation: `content_md` column exists in the DB, `react-markdown@10.1.0` + `remark-gfm@4.0.1` are installed, and the `drafter_node` in `brainstorm.py` already uses raw `ainvoke()` — making it the canonical pattern to follow for the refactored sprint planner.

The biggest implementation risk is sprint separator parsing: `---` appears in sprint content (as section dividers inside each sprint), so the separator between sprints must use a distinct pattern — `\n\n---\n\n` (double newlines on both sides) with proper regex anchoring. The existing reference documents in `plan/` define the exact target format; the prompt must include those as verbatim few-shot examples.

**Primary recommendation:** Use `\n\n---\n\n` as the inter-sprint separator in both the prompt instruction and the parser regex. Include the full `plan/sprint-1.md` content verbatim in the prompt as a format example. Apply the `drafter_node` raw-`ainvoke()` pattern directly to the sprint planner.

### Pattern 2: Sprint Separator Parsing — Safe Split Strategy

The risk: `---` is a valid horizontal rule that appears INSIDE each sprint document (the reference `plan/sprint-1.md` uses `---` between sections). The inter-sprint separator must be distinguished.

**Strategy from CONTEXT.md specifics:**
- Prompt instructs the model to use `\n\n---\n\n` (double newline on both sides) between sprints.
- Parser splits on `\n\n---\n\n` exactly.
- Within-sprint section separators are `\n\n---\n\n` too — this is the problem.

**Revised strategy:** Since `plan/sprint-1.md` uses `---` WITHOUT blank lines before/after as section dividers (inside fenced code blocks and as section breaks), the LLM must be instructed to use a DISTINCT separator between sprints. Use a unique string that cannot appear in sprint content:

```python
# Source: CONTEXT.md specifics section + analysis of plan/sprint-1.md
SPRINT_SEPARATOR = "\n\n---\n\n"

def parse_sprint_markdown(raw: str) -> list[dict]:
    """Split raw LLM output into per-sprint dicts."""
    chunks = re.split(r"\n\n---\n\n", raw.strip())
    sprints = []
    for chunk in chunks:
        chunk = chunk.strip()
        if not chunk:
            continue
        # Extract sprint_number from "# Sprint N" or "# Sprint N — ..."
        m = re.match(r"^#\s+Sprint\s+(\d+)", chunk, re.IGNORECASE)
        sprint_number = int(m.group(1)) if m else len(sprints) + 1
        # Extract goal from "## Sprint Goal\n\n<one sentence>"
        goal_match = re.search(
            r"##\s+Sprint\s+Goal\s*\n+(.+?)(?:\n|$)", chunk, re.IGNORECASE
        )
        goal = goal_match.group(1).strip() if goal_match else ""
        sprints.append({
            "number": sprint_number,
            "goal": goal,
            "content_md": chunk,
        })
    # Fallback: on complete parse failure, store raw as sprint 1
    if not sprints:
        sprints = [{"number": 1, "goal": "", "content_md": raw.strip()}]
    return sprints
```

**Prompt instruction for separator:** The sprint planner prompt must explicitly state:
> "Separate each sprint with exactly `\n\n---\n\n` (a blank line, three dashes, a blank line). Do NOT use any other separator. Do not add intro/outro text — output ONLY the sprint documents."

[VERIFIED: plan/sprint-1.md analyzed — within-sprint `---` bars appear after headers with NO surrounding blank lines, which means splitting on `\n\n---\n\n` (with double newlines) is safe. The sprint reference uses `---` after each section heading line with single blank lines. Splitting on double-newline-surrounded `---` is safe if the prompt enforces it as inter-sprint separator only.]

[ASSUMED: The LLM will reliably follow the `\n\n---\n\n` separator instruction. If not, the fallback stores all content as sprint 1.]

### Anti-Patterns to Avoid

- **Do not use `re.split(r"---", raw)`** — this splits on every `---` including those within sprint content (e.g., inside code blocks and section separators inside a sprint).
- **Do not use `response.content` from a `with_structured_output()` chain** — `with_structured_output()` returns a Pydantic model object, not an `AIMessage`. Call `ainvoke()` on the raw `llm_planner` instance, not on `structured_planner`.
- **Do not forget `"use client"`** — react-markdown@10 is ESM-only and browser-only; rendering it in a Server Component causes a module resolution error.
- **Do not use `prose` classes without `prose-invert`** — the app uses dark backgrounds; without `prose-invert`, prose text renders as near-black on dark background.
- **Do not add intro/outro text in the sprint prompt** — Groq models tend to add preamble like "Here are your sprints:". The prompt must explicitly say "Output ONLY the sprint documents with no introductory text."

## Common Pitfalls

### Pitfall 1: `---` Split False Positives
**What goes wrong:** `re.split(r"\n---\n", raw)` splits inside sprint content. Plan/sprint-1.md uses `---` between EVERY section.
**Why it happens:** The reference document uses horizontal rules as visual section separators. The LLM mimics this style inside each sprint.
**How to avoid:** Use `\n\n---\n\n` (double newlines on both sides) as the inter-sprint separator. Instruct the model explicitly. The within-sprint `---` bars in `plan/sprint-1.md` use single blank lines (not double).
**Warning signs:** After parsing, `len(sprints) >> expected_count` — you're splitting within a sprint.

### Pitfall 2: `with_structured_output` vs Raw `ainvoke`
**What goes wrong:** Calling `structured_planner.ainvoke(messages)` after removing `with_structured_output` raises `AttributeError` (structured_planner would no longer exist if deleted).
**Why it happens:** The existing `sprint_planner.py` creates `structured_planner = llm_planner.with_structured_output(SprintPlan)` at module level.
**How to avoid:** Delete `structured_planner` variable. Call `llm_planner.ainvoke(messages)` directly. Access result via `response.content`.
**Warning signs:** `ValidationError` or `AttributeError` during sprint generation.

### Pitfall 3: Groq Adds Preamble Text
**What goes wrong:** LLM prepends "Here are your sprint plans:" or similar before the first `# Sprint 1`.
**Why it happens:** Instruction-tuned Llama models default to conversational framing.
**How to avoid:** Add to prompt: "Output ONLY the sprint documents. Do not write any introductory text, summary, or explanation before or after the sprints."
**Warning signs:** First "chunk" after split has no `# Sprint` heading; regex match fails.

### Pitfall 4: max_tokens Too Low for Multi-Sprint Output
**What goes wrong:** Sprint output truncates mid-document; last sprint is incomplete.
**Why it happens:** Current `llm_planner` is configured with `max_tokens=8192` — this is already set correctly in `sprint_planner.py` but should be verified for 8-10 sprint outputs.
**Estimate:** A 10-sprint output at ~600 tokens/sprint = 6000 tokens. 8192 is sufficient. For very complex projects, 16384 would be safer.
**How to avoid:** Keep max_tokens=8192 (already correct). The model's own output limit is 32,768 tokens per the Groq docs.
**Warning signs:** Last sprint in parsed output cuts off mid-sentence.

[VERIFIED: Groq llama-3.3-70b-versatile max output tokens = 32,768 — confirmed via official Groq docs at console.groq.com/docs/model/llama-3.3-70b-versatile]

### Pitfall 5: DrUI SprintCard Renders Old Structured Shape
**What goes wrong:** After refactor, `SprintCard` still renders `sprint.user_stories`, `sprint.technical_tasks`, `sprint.definition_of_done` — which are now empty for new generations.
**Why it happens:** Old `Sprint` interface had all fields required; SprintCard renders them directly.
**How to avoid:** Update `SprintCard` to check `sprint.content_md` first. If present, render via react-markdown. If absent (legacy rows), render old Section components. Update `types/sprint.ts` to make structured fields optional.
**Warning signs:** Sprint cards show empty sections for newly generated sprints.

### Pitfall 6: Sprint Save Payload Still Uses Old Shape
**What goes wrong:** `SprintsPage.handleGenerate` sends `{sprints: accumulated}` where `accumulated` still has the old `Sprint` shape — the API route expects `content_md` field.
**Why it happens:** The PUT handler in `frontend/app/api/projects/[id]/sprints/route.ts` writes `contentMd: s.content_md ?? null` — if `content_md` is absent, it saves null.
**How to avoid:** Update `accumulatedSprintsRef` type to `Array<{number, goal, content_md}>`. Ensure SSE parsing reads `event.data.content_md`.
**Warning signs:** Sprints saved to DB have `content_md = null`; sprint page re-load shows empty cards.

### Pitfall 7: Drafter Monetization Section Appears Unconditionally
**What goes wrong:** Drafter always generates a Monetization section even for projects with no revenue model.
**Why it happens:** Prompt says "produce all sections" without conditionally omitting Monetization.
**How to avoid:** The drafter prompt must include the rule: "Omit the Monetization section entirely if the conversation data contains no `monetization` field." The ExtractedFields has `monetization: Optional[str]`; when it is None, the field is absent from `extracted` dict (due to `exclude_none=True`).
**Warning signs:** project.md for a free hobby app includes "Monetization: N/A" or placeholder.

import re

SPRINT_SEPARATOR_PATTERN = re.compile(r"\n\n---\n\n")

def _parse_sprint_markdown(raw: str) -> list[dict]:
    chunks = SPRINT_SEPARATOR_PATTERN.split(raw.strip())
    sprints = []
    for chunk in chunks:
        chunk = chunk.strip()
        if not chunk:
            continue
        # Sprint number: "# Sprint 1" or "# Sprint 1 — Goal Text"
        num_match = re.match(r"^#\s+Sprint\s+(\d+)", chunk, re.IGNORECASE)
        sprint_number = int(num_match.group(1)) if num_match else len(sprints) + 1
        # Goal: first line of ## Sprint Goal section
        goal_match = re.search(
            r"##\s+Sprint\s+Goal\s*\n+(.+?)(?:\n|$)", chunk, re.IGNORECASE
        )
        goal = goal_match.group(1).strip() if goal_match else ""
        sprints.append({"number": sprint_number, "goal": goal, "content_md": chunk})
    if not sprints:
        # Total parse failure — store raw as sprint 1 (D-04 fallback)
        return [{"number": 1, "goal": "", "content_md": raw.strip()}]
    return sprints

## Open Questions (RESOLVED)

1. **Should the sprint planner prompt include `plan/sprint-2.md` as a second example?**
   - What we know: CONTEXT.md specifics say "few-shot formatting beats prose instructions"; the canonical refs list both sprint-1.md and sprint-2.md.
   - What's unclear: Including both adds ~800 tokens of system prompt; unclear if this significantly improves consistency over just one example.
   - Recommendation: Start with sprint-1.md only (shorter prompt, faster). Add sprint-2.md if output format is inconsistent during testing.
   - **RESOLVED (Plan 01 Task 1):** Sprint planner prompt uses `plan/sprint-1.md` only as the verbatim few-shot example. `plan/sprint-2.md` is listed in canonical_refs for reference but not embedded in the prompt.

2. **Should `SprintState` be updated to accommodate the new `content_md` shape explicitly?**
   - What we know: `SprintState.sprints` is `List[dict]` — any dict shape works.
   - What's unclear: Whether adding type hints inside the TypedDict for the new shape improves IDE support enough to be worth the churn.
   - Recommendation: Leave `SprintState.sprints: List[dict]` as-is (it's already untyped dicts). The `Sprint` Pydantic models being deleted are the only type-unsafe part.
   - **RESOLVED (Plan 01 Task 2):** `SprintState.sprints` stays typed as `List[dict]`. No TypedDict change needed — the Pydantic `Sprint` and `SprintPlan` models are deleted; the raw dict shape `{number, goal, content_md}` flows through the existing `List[dict]` field without any model update.

3. **Does the `rehydrateSprints()` function in SprintsPage need a transitional path for existing rows?**
   - What we know: Existing rows have `sprintData` (structured) and may have `content_md = null`. After this phase, new rows have `content_md` and no `sprintData`.
   - What's unclear: Whether existing test projects need to be re-generated or can display legacy.
   - Recommendation: Update `SprintCard` to check `content_md` first, fall back to `sprintData` rendering. Keep `rehydrateSprints()` for loading from DB but extend it to pass `content_md` through.
   - **RESOLVED (Plan 04 Task 3):** `rehydrateSprints()` is extended with `contentMd: string | null` on `SprintRowApi` and populates `content_md: row.contentMd ?? blob?.content_md ?? ""`. SprintCard checks `sprint.content_md.trim().length > 0` and falls back to the legacy Section components when empty. No re-generation needed for existing projects.
