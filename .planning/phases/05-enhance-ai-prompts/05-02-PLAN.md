## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frontend chat → Next.js → FastAPI `/api/brainstorm` | User conversation messages cross three hops; arrive at the LLM as user input |
| `extracted` dict → drafter LLM | LLM-extracted structured fields are JSON-serialized and passed to a second LLM as input — extracted fields could embed adversarial text from the original conversation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Tampering / Information Disclosure (Prompt Injection — V5) | `backend/prompts/drafter_system.txt` (LLM consumes user-derived `extracted` JSON) | mitigate | Embed UNTRUSTED DATA guard verbatim in `drafter_system.txt` (Task 3). Verified by `grep -q "UNTRUSTED DATA" backend/prompts/drafter_system.txt`. The extractor_system.txt already operates under the same guard semantically (it's structured-output and only produces field values), so the `monetization` and `sprint_count_hint` fields cannot be injection vectors past the structured boundary. |
| T-05-06 | Information Disclosure (drafter leaking system prompt content) | drafter LLM | accept | Low risk — drafter system prompt contains only public format guidance; no secrets. |
| T-05-07 | Tampering (extractor returns malicious content for `monetization`) | downstream `drafter_system.txt` consumes `monetization` field | mitigate | Drafter prompt UNTRUSTED DATA guard covers this (Task 3). The extracted fields are JSON-serialized and treated as data by the drafter, never as instructions. |
</threat_model>
